# 00_COMMON

Общие правила для всех ролей в этом репозитории.

## Канонические термины

- `Owner` — внешний оператор системы и финальный authority source для runtime execution.
- `AGENTS.md` — bootstrap/router entrypoint, а не отдельный competing rulebook.
- `Role` — активный ролевой режим текущего прогона: `LEAD`, `ANALYST`, `CODER`, `TESTER`, `VERIFIER`, `REVIEWER`.

## Граница репозитория

- Все проектные изменения выполнять только внутри текущего репозитория, если Owner явно не задал другой scope.
- Не менять файлы вне рабочего каталога репозитория без прямого указания Owner.

## Приоритет слоёв

При конфликте применять слои сверху вниз:

1. последние явные runtime-указания Owner в текущем чате;
2. активный role-file;
3. `.agents/rules/specs/*`;
4. этот файл;
5. `AGENTS.md` как router/bootstrap;
6. `.agents/templates/*` и `.agents/skills/*`;
7. historical/evidence/advisory материалы.

Helper-layer и history/advisory layer не могут переопределять пункты 1-4.

## Общие инварианты

- Не расширять scope без явного запроса Owner.
- Прямой запрос Owner может быть текущим task contract даже без formal `IN`-тикета.
- Не подменять провал или остановку работы фиктивным `DONE` через смену цели задним числом.
- Если обязательный файл или артефакт отсутствует, зафиксировать это явно и продолжить по доступному контексту.
- Предпочитать минимально достаточные изменения и фактические формулировки.

## Что означает "open"

- Если Owner, тикет или слой правил говорят `open file`, `open report` или `open ticket`, по умолчанию это означает открыть локальный файл через `Start-Process`, а не печатать его полный markdown в чат.
- По умолчанию это значение относится только к ключевым процессным markdown-артефактам:
  - формальным тикетам `MAIL/<ROLE>/IN/*.md`;
  - формальным отчётам `MAIL/<ROLE>/OUT/*_report.md`.
- По умолчанию это значение не относится к `RUNLOGS/**`, `.txt`, другим вспомогательным логам, evidence/file packs, `.agents-runtime/00_STATE.md` и `.agents-runtime/PLANS/**`, если Owner отдельно не потребовал иное.

## Ownership process-артефактов

- `.agents-runtime/00_STATE.md` обновляет только `LEAD`.
- `.agents-runtime/PLANS/**` по умолчанию создаёт и ведёт `LEAD`, если Owner не указал иное.
- Formal `MAIL/<ROLE>/IN/**` по умолчанию создаёт `LEAD`, если Owner явно не задал другой flow.
- Каждая не-LEAD роль пишет только собственные `MAIL/<ROLE>/OUT/**`.
- Другие роли могут рекомендовать state change только в своём `OUT` или в чате, но не применяют его сами.

## Новые процессные файлы

- Если агент создаёт новый ключевой процессный файл, который должен стать частью актуального процессного слоя, агент обязан адресно выполнить `git add` именно для этого нового файла.
- По умолчанию это относится только к новым формальным тикетам `MAIL/<ROLE>/IN/*.md` и новым формальным отчётам `MAIL/<ROLE>/OUT/*_report.md`.
- По умолчанию это не относится к `RUNLOGS/**`, `.txt`, evidence/file packs и другим вспомогательным или промежуточным артефактам, если Owner отдельно не потребовал иное.

## Ожидания к результату

- Отчёты должны фиксировать факты, а не догадки.
- Если нужны длинные логи или временные диагностические артефакты, складывать их в `.agents-runtime/RUNLOGS/<ticket-id>/`.
- Для новых process-файлов использовать шаблоны из `.agents/templates/*`, если текущий тикет или rules/specs не требуют более строгую форму.
