## Что остаётся прежним для обычного file-context
- Операция по-прежнему работает только для непустого выделения.

- Текст внутри кодового блока по-прежнему берётся из фактического выделения и нормализует переводы строк к `\n`.
- Если контекст не является diff-context, итоговый формат остаётся тем же семейством LLM-блоков, что уже используется сейчас для обычного редактора.

## Спецификация для обычного file-context

### Правило определения `LANG`
Использовать следующее правило без альтернативных веток по расширению:

1. Каноническим значением является язык или тип файла, который JetBrains Platform уже определила для открытого файла в активном редакторе.
2. Если IDE даёт именно язык, в `LANG` записывается его стабильный идентификатор в нормализованном нижнем регистре.
3. Если язык как отдельная сущность недоступен, но IDE надёжно знает тип файла, в `LANG` записывается нормализованное текстовое имя этого IDE-типа.
4. Если IDE не может надёжно определить ни язык, ни тип файла, в `LANG` записывается `text`.

### Нормализация значения `LANG`
- Значение должно быть коротким, machine-friendly и стабильным между запусками.
- Предпочтительный вид: lowercase token без пробелов, например `kotlin`, `xml`, `json`, `plaintext` может быть приведён к `text`.
- Если IDE-идентификатор содержит служебные или vendor-specific вариации, они должны быть нормализованы до общеупотребимого токена, если такой токен очевиден.
- Если общеупотребимый токен неочевиден, безопасный fallback: `text`.

### Почему extension map больше не должен быть источником истины
- Расширение — слишком грубая эвристика для IDE-контекста, где платформа уже знает больше, чем имя файла.
- Файлы вроде `nuget.config` могут иметь слабое или нестандартное расширение, но IDE определяет их как XML-подобный тип и это знание качественнее для LLM.
- Поддержка вручную собранного extension-map неизбежно будет расходиться с тем, что реально показывает IDE пользователю.

### Пример для `nuget.config`
- Если JetBrains Platform определяет открытый `nuget.config` как XML/file type XML, в выводе должно быть `[LANG] xml`.
- Тот факт, что расширение у файла не `xml`, не должен менять результат.

## Спецификация для diff-context

### Когда контекст считается diff-context
Контекст считается diff-context, если действие вызвано из редактора или viewer-а, принадлежащего diff/commit UI JetBrains, и платформа позволяет надёжно получить данные diff-представления.

Практическое правило для следующей реализации:
- Сначала пытаться определить diff-context через diff/commit viewer API и связанные data keys/объекты контекста.
- Только если diff-context не обнаружен, переходить к обычному file-context.
- Не определять diff-context по косвенным эвристикам вроде имени вкладки, текста заголовка окна или наличия символов `+/-` на экране.

### Канонический формат буфера обмена для diff
Рекомендуемый формат:

`[DIFF_FILE] <path-or-file-label>`
`[DIFF_RANGE] <selected-line-range-in-view-or-unknown>`
`[LANG] diff`
`[DIFF_FORMAT] unified`
`[DIFF]`
`@@ ... @@`
`<diff lines with prefixes>`
`[/DIFF]`

Где:
- `[DIFF_FILE]` — относительный путь файла, если он надёжно известен; иначе стабильная подпись файла/изменения из viewer-а;
- `[DIFF_RANGE]` — диапазон строк в самом diff-viewer, если его можно надёжно выразить; иначе `unknown`;
- `[LANG] diff` — фиксированное значение для всего diff-context, даже если underlying file имеет собственный язык;
- `[DIFF_FORMAT] unified` — явный маркер, что тело блока представляет unified diff;
- `[DIFF] ... [/DIFF]` — отдельная секция, чтобы LLM не интерпретировала содержимое как обычный `[CODE]`.

### Почему выбрана именно эта форма
- Unified diff — самый узнаваемый для LLM и для человека формат изменений.
- Отдельные маркеры `[LANG] diff`, `[DIFF_FORMAT] unified` и `[DIFF]` убирают двусмысленность между кодом и патчем.
- Обёртка в существующем стиле квадратных маркеров сохраняет преемственность с текущим форматом плагина.
- Хранить old/new раздельно как два независимых кодовых блока хуже: теряется компактная семантика изменений и взаимное соответствие удалённых/добавленных строк.

### Что именно копировать из diff viewer
Рекомендуемое правило:
- Базовая единица копирования в diff-context — diff line, а не произвольный набор символов внутри строки.
- Если пользователь выделил фрагмент, который затрагивает несколько строк diff viewer, копируются все diff lines, которых коснулось выделение.
- Каждая строка должна сохранять свой diff-префикс:
  - `+` для добавленной строки;
  - `-` для удалённой строки;
  - пробел для context line;
  - заголовок hunk-а вида `@@ ... @@`, если он входит в выбранный диапазон или нужен для сохранения валидного локального контекста.
- Old/new не копируются двумя отдельными блоками; они объединяются в единый unified diff fragment.

### Правило про hunk header
- Если выделение уже включает `@@ ... @@`, он копируется как есть.
- Если пользователь выделил только тело изменений внутри одного hunk-а и платформа позволяет однозначно определить соответствующий `@@ ... @@`, hunk header должен быть автоматически добавлен в начало diff-блока.
- Если надёжно определить корректный hunk header нельзя, допускается копирование без него, но только если сами строки всё ещё имеют префиксы `+/-/ ` и остаются однозначно diff-представлением.

## Edge-cases и fallback-сценарии

### Обычный file-context

#### 1. IDE знает язык лучше, чем расширение
- Побеждает IDE-определение.
- Extension map не используется как override и не участвует в конфликте.

#### 2. IDE возвращает слишком общий тип вроде plain text
- Если платформа не даёт более специфичного значения, в `LANG` пишется `text`.
- Не нужно пытаться доугадывать язык по расширению поверх plain text.

#### 3. У файла нет расширения или расширение нестандартное
- Если IDE определила язык/тип, использовать его.
- Иначе `text`.

#### 4. Несохранённый, временный или служебный документ
- Если для него нет надёжного file-context с путём и IDE-язык не даёт полезного результата, поведение остаётся консервативным: не изобретать новый формат сверх уже принятого для обычного редактора.
- Если текущая реализация для такого случая уже делает fallback на plain selected text, это поведение не должно ухудшаться данным тикетом.

### Diff-context

#### 1. Неполное выделение внутри diff line
- Выделение на уровне символов должно расширяться до целых затронутых diff lines.
- Частичный символный diff в буфер не копируется, потому что он разрушает семантику unified diff.

#### 2. Выделение пересекает строки разных типов
- В результат попадают все затронутые строки в исходном порядке viewer-а.
- Тип каждой строки маркируется её diff-префиксом, без дополнительной перекодировки.

#### 3. Выделение охватывает context, removed и added строки одновременно
- Это допустимо.
- Все строки копируются в одном unified diff fragment, потому что для LLM полезно видеть локальный контекст и обе стороны изменения вместе.

#### 4. Выделение попало сразу в несколько hunk-ов
- Копируются все затронутые строки в порядке их следования.
- Для каждого hunk-а по возможности сохраняется или добавляется собственный `@@ ... @@` header.

#### 5. Viewer показывает side-by-side diff
- Формат копирования всё равно должен быть unified diff, а не две колонки.
- Реализация должна логически преобразовать выбранные строки из visual side-by-side представления в один линейный unified fragment.

#### 6. Viewer показывает только одну сторону или API отдаёт неполные данные
- Если из API можно надёжно собрать корректный unified fragment, использовать diff-format.
- Если нельзя надёжно понять соответствие old/new/context lines, не пытаться синтезировать полу-дифф из догадок.

#### 7. Diff-контекст визуально есть, но API не даёт надёжного доступа
- Не формировать кастомный diff-format на основании текста экрана.
- Если это одновременно не является обычным file-context, операция должна завершиться без специального diff-результата.
- Главный принцип: лучше не выдать diff вообще, чем выдать ложно-точный diff.

## Правило переключения режима
Использовать такой порядок принятия решения:

1. Если активный контекст надёжно распознан как diff/commit viewer и его diff-данные доступны через API, использовать diff-format.
2. Иначе, если активный контекст является обычным файловым редактором с доступным file-context, использовать существующий file-format.
3. Иначе применять уже существующее безопасное поведение действия без выдумывания нового формата и без синтетических эвристик.

Это правило важно, чтобы diff viewer не деградировал молча в `[CODE]` с `[LANG] xml|kotlin|...`, когда пользователь на самом деле копирует именно изменение.
