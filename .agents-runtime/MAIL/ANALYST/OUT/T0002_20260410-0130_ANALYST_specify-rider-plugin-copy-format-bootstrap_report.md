## Спецификация результата копирования

### Итоговый формат буфера обмена
Операция копирования должна записывать в буфер обмена единый текстовый блок ровно в таком виде:

`<path>:<lineStart>-<lineEnd>`
`<selectedTextNormalized>`

То есть:
- первая строка содержит только заголовок диапазона в формате `<path>:<lineStart>-<lineEnd>`;
- вторая и последующие строки содержат текст выделения;
- между первой строкой и текстом выделения всегда ровно один символ перевода строки `\n`;
- дополнительных префиксов, суффиксов, пустых строк до или после блока не добавляется.

### Правило для `path`
- `path` — это относительный путь файла от корня текущего проекта/репозитория, в котором расположен Rider-плагин.
- Разделитель в `path` — прямой слеш `/`, независимо от ОС.
- Если выделение сделано в файле внутри текущего репозитория, `path` обязан быть вычислен относительно его корня.
- Абсолютный путь диска, буква диска и machine-specific префиксы в `path` не допускаются.
- Если активный редактор не привязан к физическому файлу внутри текущего репозитория, операция не должна формировать альтернативный формат. В этом случае плагин должен завершить действие без записи нового содержимого в буфер обмена. Это ограничение нужно сохранить и в реализации.

Примеры корректного `path`:
- `src/main/java/com/acme/MyAction.java`
- `.agents-runtime/MAIL/ANALYST/IN/T0002_20260410-0130_ANALYST_specify-rider-plugin-copy-format-bootstrap.md`

### Правило для `lineStart`
- `lineStart` — номер первой строки файла, в которую попадает хотя бы один символ выделения.
- Нумерация строк 1-based.
- Если выделение начинается с середины строки, `lineStart` всё равно равен номеру этой строки.
- Если выделение начинается ровно с первого символа строки, используется номер этой строки.

### Правило для `lineEnd`
- `lineEnd` — номер последней строки файла, которая должна считаться частью копируемого выделения.
- Нумерация строк 1-based.
- Если выделение заканчивается на середине строки, `lineEnd` равен номеру этой строки.
- Если выделение заканчивается ровно на границе между строками и последний символ выделения — перевод строки предыдущей строки, то последняя пустая/следующая строка в диапазон не включается.
- Если выделение включает символы на следующей строке, даже один символ, `lineEnd` равен номеру этой следующей строки.

### Правило для `selectedTextNormalized`
- В текстовую часть попадает фактический выделенный текст из редактора.
- Текст сохраняется посимвольно, без добавления или удаления внутренних символов, кроме нормализации переводов строк к `\n`.
- Если исходное выделение содержит завершающий перевод строки, он сохраняется в `selectedTextNormalized`.
- Если исходное выделение не содержит завершающий перевод строки, он не добавляется.
- Текст после заголовка начинается сразу после единственного разделительного `\n` между заголовком и телом.

## Ожидаемое поведение по сценариям

### 1. Пустое выделение
- Если в редакторе нет выделения и есть только caret, операция не должна вычислять искусственный диапазон строки/слова.
- Буфер обмена не должен изменяться.
- Это поведение выбрано как минимально достаточное и не расширяет scope до “copy current line”.

### 2. Выделение в пределах одной строки
- Если выделение затрагивает часть одной строки, результат:
  - `lineStart = lineEnd = номер этой строки`;
  - в текстовой части копируется только выделенный фрагмент.

Пример:

Исходный файл:
- строка 12: `const answer = 42;`

Выделено: `answer`

Результат в буфере:
`src/example.ts:12-12`
`answer`

### 3. Выделение нескольких полных строк
- Если выделение начинается в одной строке и заканчивается в другой, диапазон покрывает все строки, в которых есть выделенные символы.
- Текстовая часть содержит выделение в исходном виде, включая внутренние переводы строк.

### 4. Частичное выделение первой и/или последней строки
- Если первая строка выделена частично, `lineStart` всё равно указывает на неё.
- Если последняя строка выделена частично, `lineEnd` всё равно указывает на неё.
- Плагин не расширяет текст до полных строк; в буфер попадает только фактически выделенный текст.

### 5. Выделение заканчивается переводом строки
- Если пользователь выделил текст так, что последним символом попал перевод строки конца строки `N`, тогда:
  - текстовая часть заканчивается `\n`;
  - `lineEnd = N`, а не `N + 1`, если на строке `N + 1` не выделено ни одного символа.

Пример:
- выделен текст строк 20-21 вместе с переводом строки после строки 21;
- результат заголовка: `<path>:20-21`;
- текстовая часть содержит содержимое строк 20-21 и завершающий `\n`.

### 6. Выделение начинается или заканчивается ровно на границе строки
- Начало выделения на границе строки трактуется как начало соответствующей строки.
- Конец выделения на границе строки трактуется по фактически включённому последнему символу:
  - если последний включённый символ принадлежит строке `N`, то `lineEnd = N`;
  - следующая строка не включается только из-за позиции caret после перевода строки.

## Edge-cases по типам вкладок и файлов
- Обычные текстовые файлы внутри текущего репозитория проекта: формат применяется как описано выше.
- Файлы с любым расширением, открытые как текст в редакторе Rider: формат применяется одинаково; расширение не влияет на результат.
- Новые несохранённые вкладки без пути на диске: операция не изменяет буфер обмена, потому что `path` нельзя однозначно вычислить относительно корня текущего репозитория.
- Вкладки системных preview/diff/settings/terminal/служебных окон Rider, где нет обычного файла проекта: операция не изменяет буфер обмена.
- Файлы вне текущего репозитория проекта, даже если Rider их открыл: операция не изменяет буфер обмена, потому что реализация по текущему scope не должна работать за пределами корня этого репозитория.

## Ограничения для следующего шага реализации
- Любая разработка по этому тикету и следующему CODER-тикету выполняется только внутри текущего корня данного репозитория.
- Изменения файлов вне текущего корня данного репозитория запрещены.
- Реализация не должна требовать правок в соседних репозиториях, глобальных IDE-настройках или внешних каталогах.
- Текущий физический путь репозитория может меняться при переносе проекта; формат копирования и правила вычисления `path` должны оставаться корректными после такого переноса без изменения спецификации.
- Если для конкретной вкладки/редактора невозможно однозначно получить `path` внутри текущего репозитория, операция должна завершаться без изменения буфера обмена.
