## Контекст

Текущий плагин умеет копировать выделение из обычного редактора в LLM-ориентированном формате:

`[FILE] <path>`
`[LINES] <lineStart>-<lineEnd>`
`[LANG] <language-or-text>`
`[CODE]`
`<selectedTextNormalized>`
`[/CODE]`

После практического использования выявлены два новых требования:

1. Для некоторых файлов текущая эвристика языка по расширению даёт сомнительный результат. Пример: `nuget.config` определяется как `text`, хотя по смыслу ожидается как минимум `xml`.
2. Пользователь хочет копировать изменения из diff/commit viewer так, чтобы LLM однозначно понимала, что перед ней именно diff, а не просто кусок кода. Нужен отдельный формат и чёткие правила поведения для diff-контекста.

Нужно подготовить спецификацию, по которой `CODER` сможет реализовать это без догадок.

## Scope

- Уточнить, как определять `LANG` для обычного файлового контекста: что именно считать источником истины, если extension-map и реальный тип файла расходятся.
- Уточнить целевой формат копирования для diff-контекста.
- Описать, как различать обычный editor-контекст и diff viewer/commit viewer.
- Зафиксировать edge-cases и fallback-поведение.
- Не переходить к реализации.

## Вопросы, которые нужно закрыть

1. Какой источник языка считать каноническим:
   - JetBrains file type / language API;
   - расширение файла;
   - fallback на `text`.
2. Какой формат diff-копирования оптимален для LLM:
   - unified diff / patch-подобный;
   - структурированный кастомный формат с маркерами;
   - другой вариант.
3. Что именно копировать из diff viewer:
   - только выделенный фрагмент;
   - весь hunk;
   - old/new одновременно;
   - только current/new side;
   - как маркировать added/removed/context lines.
4. Что делать, если в diff viewer выделение неполное или пересекает разные типы строк.
5. Как вести себя, если diff-контекст есть, но его нельзя надёжно извлечь через доступные API.
