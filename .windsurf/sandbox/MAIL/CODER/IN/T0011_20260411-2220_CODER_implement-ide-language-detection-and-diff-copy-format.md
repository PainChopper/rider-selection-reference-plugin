## Контекст

`ANALYST` завершил спецификацию `T0010` по двум связанным улучшениям:

1. В обычном file-context значение `LANG` должно определяться не ручной map-эвристикой по расширению, а канонически через язык/тип файла, который уже определила JetBrains Platform.
2. В diff/commit viewer нужен отдельный режим копирования, в котором результат однозначно представляется как diff для LLM, а не как обычный кодовый фрагмент.

Сейчас плагин уже умеет копировать обычный file-context в LLM-структуре, но:
- `LANG` для файлов вроде `nuget.config` определяется некачественно;
- diff-контекст не имеет отдельного представления.

Дополнительное наблюдение из пользовательского прогона:
- если пользователь стоит в правой (current/new) версии diff и нажимает `F4`, Rider открывает обычное окно редактирования самого файла;
- это может быть полезно как ориентир для следующей реализации: правая сторона diff связана с реальным file-context, и этот факт можно учитывать при выборе API-стратегии или fallback-поведения, но без деградации diff-mode в обычный file-format, если diff-данные доступны надёжно.

Нужно реализовать спецификацию `T0010` без расширения scope.

## Scope

- Заменить текущее определение `LANG` на IDE-based определение через JetBrains Platform language/file type API.
- Для обычного file-context сохранить текущий LLM-формат, но с новым правилом определения `LANG`.
- Добавить отдельный diff-mode для diff/commit viewer.
- Для diff-mode формировать unified diff внутри явных маркеров, как зафиксировано в `T0010`.
- Сохранить существующее безопасное поведение там, где контекст или данные нельзя надёжно извлечь.

## Целевое поведение

### 1. Обычный file-context
Для обычного редактора формат остаётся:

`[FILE] <path>`
`[LINES] <lineStart>-<lineEnd>`
`[LANG] <ide-detected-language-or-text>`
`[CODE]`
`<selectedTextNormalized>`
`[/CODE]`

Правило для `LANG`:
- источник истины — язык/тип файла, который уже определила JetBrains Platform;
- если IDE даёт язык, использовать его нормализованный lower-case идентификатор;
- если языка нет, но есть надёжный file type, использовать его нормализованное machine-friendly имя;
- если IDE не даёт надёжного ответа, использовать `text`;
- ручной extension-map больше не должен быть главным источником истины.

Пример ожидаемого результата:
- `nuget.config` при IDE-определении как XML должен давать `[LANG] xml`, а не `text`.

### 2. Diff-context
Если action вызвана в diff/commit viewer и diff-данные доступны надёжно через API, использовать формат:

`[DIFF_FILE] <path-or-file-label>`
`[DIFF_RANGE] <selected-line-range-in-view-or-unknown>`
`[LANG] diff`
`[DIFF_FORMAT] unified`
`[DIFF]`
`@@ ... @@`
`<diff lines with prefixes>`
`[/DIFF]`

Правила:
- базовая единица копирования — diff line, а не частичный набор символов внутри строки;
- частичное выделение по символам должно расширяться до затронутых diff lines;
- строки сохраняют свои diff-префиксы (`+`, `-`, ` `);
- если hunk header `@@ ... @@` уже входит в выделение, копировать его;
- если header не выделен, но его можно надёжно определить для выбранного фрагмента, добавить его;
- если корректный diff fragment надёжно собрать нельзя, не синтезировать fake diff.

## Правило переключения режима

1. Если контекст надёжно распознан как diff/commit viewer и diff-данные доступны через API, использовать diff-format.
2. Иначе, если это обычный file-context, использовать обычный file-format.
3. Иначе использовать уже существующее безопасное поведение action без эвристического парсинга экрана.

Примечание:
- наличие перехода по `F4` из правой стороны diff в обычный редактор файла не отменяет приоритет diff-mode;
- если action вызвана именно в diff viewer и API даёт надёжный diff-context, нужно предпочитать diff-format;
- открытие файла по `F4` следует рассматривать как полезную подсказку о связи diff-side с реальным файлом, а не как основание автоматически подменять diff-контекст обычным `[FILE]/[CODE]` форматом.
